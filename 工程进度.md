# Entry Logic Research 工程状态

生成日期：2026-06-22  
范围：Quant Replay Collector 的 entry logic research 真实使用版本。本文只记录工程状态、目标、遗留问题和下一步计划，不是营销材料，也不是收益承诺。

## 当前结论

当前版本可以开始小样本真实使用。适合做人工标注、复标队列、context features、outcome labels、purged chronological split、prototype / PU similarity scoring、实验 manifest、Markdown / JSON 报告和可选导出。

它还不适合被宣传成自动交易系统、AI 预测买点、已验证盈利策略或成熟机器学习平台。`review_queue` 只是人工复标队列，不是开仓列表。

## 工程目标

项目目标已经从 GitHub 展示项目转向用户本人长期使用的统计学习研究工具。

entry logic research 要解决的问题是：把用户在深 V 反转做多场景中的主观开仓判断，沉淀成可复现、可审计、可改进的研究流程。

边界很清楚：

- 标签是 `human_decision`，不是 `future_return`。
- `human_decision` 只允许 `ENTRY`、`REJECT`、`UNCERTAIN`、`UNLABELED`。
- `UNLABELED` 不是负样本，未开仓样本也不能自动当成 `REJECT`。
- `entry_context_features` 只使用决策时可见数据。
- `entry_outcome_labels` 单独保存，只做后验分析和报告诊断。
- 模型输出只能解释为相似度或 setup 置信度，不能叫 `buy_signal`、`sell_signal` 或 `trade_signal`。
- 金融时间序列不能随机切分，默认走时间顺序切分，并支持 purge、embargo 和 episode 检查。

## 已完成的工程进度

### 研究数据底座

已新增或加固 raw kline 质量检查、稳定 `candle_id`、`data_hash` / `data_version`、缺口和重复 K 线检查。研究模块可以把数据质量 warning 写入报告或 manifest。

仍要注意：原始行情数据必须保持只读语义，研究产物不能覆盖 raw kline。

### 标注系统

`EntryAnnotation` 已支持长期标注需要的字段：

- `setup_bar_index`
- `decision_bar_index`
- `setup_bar_time`
- `decision_bar_time`
- `annotation_version`
- `updated_at`
- `is_active`
- `superseded_by`

旧字段 `bar_index` 保留兼容，语义等同 `decision_bar_index`。

SQLite 层已支持 `entry_annotations` 和 `entry_annotation_history`。同一个 observation 同一时间只允许一个 active annotation。用户把 `ENTRY` 改成 `REJECT` 或 `UNCERTAIN` 时，当前策略是 update-in-place，并把旧版本写入 history。

### 候选机会集和复标队列

`entry_observation_universe` 已生成宽松候选观察点。它只负责挑出值得人工看的位置，不代表可以开仓。

关键字段包括：

- `observation_id`
- `setup_bar_index`
- `decision_bar_index`
- `feature_cutoff_bar_index`
- `candle_id`
- `candidate_source`
- `candidate_reason`
- `data_version`

`active_label_selection` 已支持：

- `high_similarity`
- `uncertain`
- `diverse`

review queue 会排除已有 active annotation 的样本。同分数按稳定字段排序，避免每次结果漂移。

### 特征和标签工程

`entry_context_features` 已引入 `FeatureSpec` 和 `FeatureQualityReport`。每行特征都有 `feature_cutoff_bar_index` 和 `feature_timing_policy`。

`NEXT_BAR_CONFIRMATION` 的语义已经明确：

- `allow_confirmation_bar=False`：只允许看到 setup bar，cutoff 是 `setup_bar_index`。
- `allow_confirmation_bar=True`：允许看到确认 K 线，cutoff 是 `decision_bar_index`。

`entry_outcome_labels` 已引入 `LabelSpec`，支持动态 horizons、手续费/滑点字段、`same_bar_policy` 和 horizon 不足处理。默认同一根 K 线同时触发 TP / SL 时按 `stop_loss_first`，这是保守假设。

### 时间序列验证

`temporal_validation` 已升级成可供 entry logic experiment 调用的验证框架。它支持：

- `SplitResult`
- `WalkForwardSplitResult`
- `purge_train_against_eval`
- `apply_embargo_against_eval`
- `build_purged_chronological_split`
- `build_purged_walk_forward_splits`
- `assign_episode_id`
- `validate_split_integrity`

`entry_logic_experiment` 默认使用 `purged_chronological`。train / validation / test 只来自已标注的 `ENTRY / REJECT`。`UNCERTAIN` 默认排除训练和评估，`UNLABELED` 只在模型训练后用于 scoring 和 review queue。

validation 用于 threshold tuning。test 只做 frozen evaluation。

### 评分、PU 和实验闭环

已完成轻量 prototype similarity 和 PU-style ranking。当前实现只用 pandas / numpy，不引入 sklearn、torch、tensorflow、xgboost。

实验 runner 的固定流程是：

1. 标准化 annotation。
2. 构造已标注数据集，只保留 `ENTRY / REJECT`。
3. 构造 `UNLABELED` scoring 数据集。
4. 做 purged chronological split。
5. 只在 train 上拟合 ENTRY prototype。
6. 在 validation 上调 threshold。
7. 在 test 上冻结评估。
8. 给 `UNLABELED` 打 `human_entry_similarity`。
9. 生成 review queue。
10. 保存 manifest、metrics、report 和 artifact paths。

`exporter` 已新增可选 entry logic 导出，同时保持旧 CSV / JSON / Markdown / Parquet 导出语义。data dictionary 明确写清：

## 仍然存在的问题

### 1. 真实样本量还不足

代码链路已经能跑，但没有足够真实人工标注样本。现在任何 scoring、threshold、ENTRY vs REJECT 差异、PU ranking 都只能当作流程验证，不能当作策略结论。

短期目标应该是积累第一批稳定标注，而不是优化模型。

### 2. UI 只够最小使用

当前 UI 入口能生成报告和 review queue，但还不是高效率标注台。每日真实使用时，用户最需要的是快速跳转、快捷键、reason_tags、note、上一条 / 下一条、保存后不全量阻塞刷新。

这些能力已有雏形，但还需要用真实标注流程打磨。

### 3. 数据版本和实验版本还需要形成习惯

工程上已经有 `data_hash`、`feature_version`、`label_version`、`split_config`、`model_config` 和 artifact paths。问题不在代码，而在使用流程：每次实验必须保存 manifest，不能只看临时输出。

如果用户手工改数据、改标注、改 feature spec 后没有生成新实验目录，后续对比会失真。

### 4. 数据质量 warning 还没有成为硬门槛

缺口、重复 K 线、乱序、OHLC 不合法、volume 缺失等问题可以被报告，但小样本研究阶段还没有统一规定哪些 warning 必须阻断实验。

长期真实使用时，这些规则需要从 warning 分级成 gate。

### 5. 统计诊断不能替代验证

偏度、超额峰度、分位数、drift、ACF、Ljung-Box、ENTRY vs REJECT 差异都已经有模块支持，但它们只能帮助理解用户判断边界。

它们不能证明未来收益，也不能说明某个候选规则可以直接交易。

### 6. 复标队列仍然依赖用户反馈闭环

`review_queue` 现在可以排序，但它的价值必须来自用户复标后的反馈。没有持续复标，active learning 只是排序表。

下一阶段要重点看：高相似样本被 REJECT 的原因、低相似样本被 ENTRY 的原因、UNCERTAIN 样本是否反复集中在某些形态。

### 7. 提交范围需要整理

当前工作区同时包含 entry logic research、UI、备份、自检、主题和测试文件。提交前需要拆分或至少明确 scope。

不能把 `.scratch/`、`dist/`、本地数据库、缓存、performance reports 放进提交。

## 下一步改进计划

### 阶段 0：冻结可用基线

目标：把当前真实研究版本固定下来，避免继续混入新功能。

动作：

- 按文件显式 stage，禁止 `git add .`。
- 排除 `.scratch/`、`dist/`、数据、日志、缓存和 performance reports。
- 重跑完整 gate。
- 写清 commit message：强调 entry logic research 是研究工具，不是交易系统。

完成标准：

- 全量测试通过。
- clean release 通过。
- `git diff --check` 通过。
- 提交范围可解释。

### 阶段 1：小样本真实标注

目标：先收集可用样本，不优化模型。

建议规模：

- 先做 50 到 100 个候选 observation。
- 每个样本都记录 `ENTRY / REJECT / UNCERTAIN`、confidence、reason_tags、note。
- 对 `NEXT_BAR_CONFIRMATION` 样本明确 setup bar 和 decision bar。

每周检查：

- ENTRY / REJECT / UNCERTAIN 比例。
- reason_tags 是否过散。
- 是否有重复 active annotation。
- review queue 是否真的提高复标效率。

### 阶段 2：把数据质量变成研究 gate

目标：防止脏 K 线污染实验结论。

动作：

- 对缺口、重复、乱序、OHLC 非法、volume 缺失设置严重级别。
- 把严重数据质量问题写入 manifest warnings。
- 对关键问题直接阻断 experiment runner。
- 明确多周期 anchor 规则，避免 1m / 5m / 15m 混用时产生对齐偏差。

### 阶段 3：打磨每日标注 UI

目标：减少人工标注摩擦。

动作：

- review queue 点击后跳转到 `decision_bar_index`。
- 同屏显示 setup bar、decision bar 和关键 context features。
- 支持 E / R / U、1-5 confidence、N / B 上下条。
- 保存后局部刷新 queue 状态，不阻塞主线程。
- 标注失败时显示错误，不丢 note。

### 阶段 4：固定每周复盘流程

目标：让研究变成周期性反馈，而不是一次性实验。

每周报告至少看：

- 本周新增标注数量。
- ENTRY / REJECT / UNCERTAIN 数量变化。
- reason_tags 排名。
- ENTRY vs REJECT 关键特征差异。
- human_entry_similarity 分布变化。
- review_queue 完成率。
- 高相似但被 REJECT 的样本。
- 低相似但被 ENTRY 的样本。
- outcome labels 的后验风险摘要。

### 阶段 5：再考虑更复杂模型

只有满足这些条件后，才值得考虑更复杂模型：

- 标注数量足够。
- 标签定义稳定。
- 数据质量 gate 稳定。
- 时间切分和 manifest 可复现。
- 当前 prototype / PU baseline 的误差模式已经清楚。

在这之前，不引入 sklearn、torch、tensorflow、xgboost。

## 不能对外宣传的点

这些说法不能写进 README、release note 或项目介绍：

- AI 预测买点。
- 自动生成交易信号。
- 策略已证明盈利。
- 样本内收益代表未来收益。
- review queue 是开仓列表。
- `human_entry_similarity` 是买入概率。
- PU score 是未来上涨概率。

可接受的说法是：

- 学习用户开仓判断边界。
- 把主观交易复盘结构化。
- 用人工标注、时间序列验证和半监督排序提高研究效率。
- 输出相似候选和复标队列，不构成投资建议。

## 当前工程判断

当前版本已经越过“展示 demo”阶段，可以进入小样本真实使用。下一步不该继续堆模块，而是先用真实数据跑一轮完整工作流。

最重要的不是模型分数，而是三件事：

1. 用户是否能稳定标出 `ENTRY / REJECT / UNCERTAIN`。
2. 每次实验是否能通过 manifest 复现。
3. review queue 是否真的减少无效复标时间。

这三件事跑通后，再谈更强的统计学习方法。
